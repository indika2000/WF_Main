# Node.js API Gateway - Planning Document

## 1. Purpose

The API Gateway is the single entry point for all mobile app requests. It handles:
- **Authentication** — Firebase ID token verification
- **Authorization** — JWT generation with embedded permissions
- **Routing** — proxy requests to appropriate backend services
- **Rate limiting** — per-user and per-IP throttling
- **Health aggregation** — unified health endpoint for all services

The gateway does **not** contain business logic — it authenticates, authorizes, and routes.

## 2. Tech Stack

| Component | Technology |
|-----------|-----------|
| Runtime | Node.js 20 LTS |
| Framework | Express.js |
| Auth | Firebase Admin SDK |
| JWT | jsonwebtoken (HS256) |
| Proxy | http-proxy-middleware |
| Rate Limiting | express-rate-limit + rate-limit-redis |
| Testing | Jest + supertest |

## 3. Middleware Stack

Every request passes through middleware in order:

```
Request
  │
  ├─ 1. Request Logger         → Log method, path, IP
  ├─ 2. CORS                   → Allow configured origins
  ├─ 3. Rate Limiter           → Per-IP throttle (100 req/min default)
  ├─ 4. Body Parser            → JSON body parsing
  ├─ 5. Auth Middleware         → Firebase token verification (skip for public routes)
  │     ├─ Extract Bearer token from Authorization header
  │     ├─ Verify via Firebase Admin SDK
  │     ├─ Attach user info to req.user (uid, email)
  │     └─ On failure → 401
  ├─ 6. Permission Middleware   → Fetch permissions, generate JWT (optional per route)
  │     ├─ Call Permissions Service: GET /permissions/{uid}
  │     ├─ If new user → POST /permissions/{uid} (create defaults)
  │     ├─ Generate internal JWT with permissions
  │     └─ Attach to req.internalToken
  └─ 7. Route Handler / Proxy  → Handle or forward to backend service
```

## 5. Service Proxy Configuration

> **Implementation note:** The actual implementation uses `pathPrefix` (prepended in the `proxyReq` handler) instead of `pathRewrite`. Express strips the router mount prefix, so the proxy receives only the sub-path. The `pathPrefix` restores the backend route prefix. See `services/node-gateway/src/config/services.js`.

```javascript
const SERVICE_CONFIG = {
  permissions: {
    url: process.env.PERMISSIONS_SERVICE_URL || 'http://permissions:5003',
    timeout: 10000,
    pathPrefix: '/permissions',
  },
  commerce: {
    url: process.env.COMMERCE_SERVICE_URL || 'http://commerce:3004',
    timeout: 30000,
    pathPrefix: '',  // Commerce routes are /cart/*, /checkout/*, etc. — no common prefix
  },
  images: {
    url: process.env.IMAGE_SERVICE_URL || 'http://image-service:5001',
    timeout: 60000,
    pathPrefix: '/images',
  },
  llm: {
    url: process.env.LLM_SERVICE_URL || 'http://llm-service:5000',
    timeout: 120000,
    pathPrefix: '',  // LLM routes are /generate/*, /providers/* — no common prefix
  },
  chat: {
    url: process.env.LLM_SERVICE_URL || 'http://llm-service:5000',
    timeout: 120000,
    pathPrefix: '/chat',  // Chat routes are /chat/* on the LLM service
  },
};
```

### 5.1 Commerce Webhook Auth Bypass

Commerce webhooks (called by Stripe directly) bypass JWT auth. The gateway proxy uses `next('route')` to skip auth for `/webhooks` paths, falling through to an unauthenticated proxy:

```javascript
// Authenticated commerce routes
router.use('/commerce', (req, res, next) => {
  if (req.path.startsWith('/webhooks')) return next('route');
  authMiddleware(req, res, next);
}, permissionsMiddleware, createServiceProxy('commerce', SERVICE_CONFIG.commerce));

// Unauthenticated fallback (webhooks only reach here)
router.use('/commerce', createServiceProxy('commerce', SERVICE_CONFIG.commerce));
```

### 5.2 fixRequestBody for POST Proxying

`express.json()` consumes the body stream before the proxy can forward it. All proxy routes call `fixRequestBody(proxyReq, req, res)` in the `proxyReq` handler to re-serialize `req.body` onto the proxy request.

## 6. JWT Token Structure

```javascript
// Generated by gateway after Firebase auth + permission fetch
{
  uid: "firebase_uid_123",
  email: "user@example.com",
  role: "user",                    // user | admin | moderator
  is_premium: false,
  permissions: {
    ad_free: false,
    premium_features: false,
    ai_text_generation: true,
    ai_image_generation: false,
    // ...all permission flags
  },
  subscription_tier: "free",
  iat: 1709312400,
  exp: 1709316000                  // 1 hour expiry
}
```

Backend services verify this JWT instead of calling Firebase for every request.

## 7. Auth Token Exchange Flow

```
Mobile App
    │
    POST /api/auth/token
    │  Header: Authorization: Bearer <firebase_id_token>
    │
    Gateway:
    ├─ Verify Firebase ID token
    ├─ Extract uid, email from decoded token
    ├─ GET /permissions/{uid} from Permissions Service
    │   ├─ 200 → use existing permissions
    │   └─ 404 → POST /permissions/{uid} (create new user with defaults)
    ├─ GET /subscriptions/{uid} from Permissions Service
    ├─ Generate internal JWT (1hr expiry)
    └─ Return: { token, user: { uid, email, permissions, subscription } }
```

## 8. Rate Limiting Strategy

| Scope | Limit | Window | Apply To |
|-------|-------|--------|----------|
| Global (per IP) | 100 requests | 1 minute | All routes |
| Auth endpoints | 10 requests | 1 minute | `/api/auth/*` |
| AI generation | 5 requests | 1 minute | `/api/llm/generate/*` |
| Image upload | 10 requests | 1 minute | `/api/images/upload` |
| Webhook (Stripe) | 50 requests | 1 minute | `/api/commerce/webhooks/*` |

Rate limit state stored in Redis for consistency across gateway restarts.

## 9. Error Response Format

The gateway normalizes all errors to a consistent format:

```json
{
  "success": false,
  "message": "Authentication required",
  "error_code": "AUTH_REQUIRED",
  "status": 401
}
```

Proxy errors (backend service down):
```json
{
  "success": false,
  "message": "Service temporarily unavailable",
  "error_code": "SERVICE_UNAVAILABLE",
  "status": 503
}
```

## 12. Security

- Firebase Admin SDK verifies tokens cryptographically (no custom verification)
- JWT secret stored in environment variable, never in code
- Rate limiting prevents brute force and DDoS
- CORS restricted to known origins in production
- Request body size limited (10MB for image uploads, 1MB default)
- Proxy removes sensitive headers before forwarding
- Stripe webhooks bypass auth middleware but verify webhook signature
