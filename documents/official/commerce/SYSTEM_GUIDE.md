## Data Models

### CartItem

Represents a single line item within a user's cart.

| Field | Type | Notes |
|-------|------|-------|
| `item_id` | string | Unique identifier for the item |
| `item_type` | enum | `subscription`, `one_time`, or `pack` |
| `name` | string | Display name of the item |
| `description` | string | Optional description |
| `quantity` | int | Number of units |
| `unit_price` | float | Price per single unit |
| `metadata` | dict | Arbitrary key-value data (e.g. tier, pack_id) |

### Cart

Persisted as a JSON string in Redis. The service owns all price calculations — client-supplied prices are never trusted.

| Field | Type | Notes |
|-------|------|-------|
| `user_id` | string | Owner of the cart |
| `items` | CartItem[] | List of cart line items |
| `subtotal` | float | Sum of (unit_price * quantity) for all items |
| `tax` | float | Always 0 (not currently calculated) |
| `shipping` | float | Always 0 (not currently calculated) |
| `total` | float | subtotal + tax + shipping |
| `updated_at` | datetime | Last mutation timestamp |

**Redis key:** `cart:{user_id}`
**TTL:** 7 days (reset on every write)

### CommerceProfile

Stored in MongoDB collection `commerce_profiles`. One record per user.

| Field | Type | Notes |
|-------|------|-------|
| `user_id` | string | Unique identifier (indexed, unique) |
| `stripe_customer_id` | string | Stripe Customer ID (sparse unique index) |
| `default_payment_method_id` | string | Stripe Payment Method ID |
| `addresses` | Address[] | List of saved addresses |
| `created_at` | datetime | Record creation timestamp |
| `updated_at` | datetime | Last update timestamp |

### Address

Embedded within `CommerceProfile.addresses`.

| Field | Type | Notes |
|-------|------|-------|
| `id` | string | UUID v4 generated at creation |
| `label` | string | Human-readable label (e.g. "Home", "Work") |
| `line1` | string | Street address line 1 |
| `line2` | string | Optional second address line |
| `city` | string | City |
| `state` | string | State or province |
| `postal_code` | string | ZIP or postal code |
| `country` | string | ISO country code, defaults to `US` |
| `is_default` | bool | Whether this is the user's default address |

### Order

Stored in MongoDB collection `orders`.

| Field | Type | Notes |
|-------|------|-------|
| `order_id` | string | Unique ID in format `ORD-YYYYMMDDHHMMSS-XXXXXX` |
| `user_id` | string | Owner of the order |
| `stripe_payment_intent_id` | string | Associated Stripe PaymentIntent |
| `stripe_subscription_id` | string | Associated Stripe Subscription (if applicable) |
| `order_type` | enum | `one_time` or `subscription` |
| `status` | enum | See status table below |
| `items` | OrderItem[] | Snapshot of cart items at time of order |
| `subtotal` | float | Cart subtotal at checkout |
| `tax` | float | Tax amount |
| `shipping` | float | Shipping amount |
| `total` | float | Final charged amount |
| `shipping_address` | Address | Snapshot of shipping address (if provided) |
| `payment_method_summary` | dict | Masked payment method info (last4, brand) |
| `metadata` | dict | Arbitrary metadata |
| `created_at` | datetime | Order creation timestamp |
| `updated_at` | datetime | Last status update timestamp |

**Order Status Values:**

| Status | Description |
|--------|-------------|
| `pending` | Order initiated but payment not yet confirmed |
| `confirmed` | Payment succeeded, order accepted |
| `processing` | Order is being fulfilled |
| `completed` | Fulfillment complete |
| `failed` | Payment failed |
| `refunded` | Payment refunded |

**Order ID format:** `ORD-YYYYMMDDHHMMSS-XXXXXX` where `XXXXXX` is a random alphanumeric suffix. Generated by `generate_order_id()` in `models/orders.py`.

### SubscriptionRecord

| Field | Type | Notes |

| `updated_at` | datetime | Last update timestamp |

## Cart System (Redis)

### Storage

Each cart is stored as a serialized JSON string in Redis:

- **Key:** `cart:{user_id}`
- **TTL:** 7 days, refreshed on every write operation

### Price Integrity

The service recalculates totals server-side on every cart mutation via `_recalculate_totals()`. Client-submitted prices are ignored — the service always computes `subtotal`, `tax`, `shipping`, and `total` from stored `unit_price` values.

### Item Merging

When a `POST /cart/{user_id}/items` request includes an `item_id` that already exists in the cart, the quantities are **merged** (added together) rather than creating a duplicate entry.

### Quantity Removal

When a `PATCH /cart/{user_id}/items/{item_id}` request sets `quantity` to `0`, the item is automatically removed from the cart rather than stored with zero quantity.

## Checkout Flow

Checkout is a three-step process. Each step must be called in sequence.

### Step 1: Validate — `POST /checkout/{user_id}/validate`

Performs pre-checkout validation:
- Confirms the cart is not empty
- Confirms the cart total is greater than 0

Returns a success response or an error indicating what is invalid.

### Step 2: Create Payment — `POST /checkout/{user_id}/create-payment`

Prepares the payment with Stripe:

1. **get_or_create_customer:** Looks up the user's `CommerceProfile` for an existing `stripe_customer_id`. If none exists, creates a new Stripe Customer and persists the ID to `commerce_profiles`.
2. **PaymentIntent creation:** Creates a Stripe PaymentIntent with the cart total (converted to integer cents), with `automatic_payment_methods` enabled.
3. **EphemeralKey creation:** Creates a Stripe EphemeralKey scoped to the customer, for use by the mobile SDK.

**Returns:**
```json
{
  "client_secret": "pi_xxx_secret_xxx",
  "ephemeral_key": "ek_xxx",
  "customer_id": "cus_xxx",
  "payment_intent_id": "pi_xxx",
  "amount": 999
}
```

### Step 3: Confirm — `POST /checkout/{user_id}/confirm`

Finalizes the order after the client-side payment completes:

1. Retrieves the PaymentIntent from Stripe.
2. Verifies the PaymentIntent status is `"succeeded"`.
3. Snapshots all current cart items into a new `Order` document (status: `"confirmed"`).
4. Retrieves payment method details to store a masked summary (e.g. last 4 digits, card brand).
5. Clears the cart from Redis.

## Subscription Lifecycle

Valid subscription tiers: `premium`, `ultra`

### Create — `POST /subscriptions/{user_id}/create`

1. Validates that the requested tier is `premium` or `ultra`. Returns `400 INVALID_TIER` if not.
2. Checks that the user does not already have an active subscription. Returns `409 ACTIVE_SUBSCRIPTION_EXISTS` if one exists.
3. Creates a Stripe Subscription with `payment_behavior: "default_incomplete"` so it starts without an immediate charge.
4. Saves a `SubscriptionRecord` to MongoDB with status `"pending"`.
5. The subscription transitions to `"active"` asynchronously when the `customer.subscription.created` webhook is received.

### Cancel — `POST /subscriptions/{user_id}/cancel`

- Sets `cancel_at_period_end: true` on the Stripe Subscription (user retains access until the billing period ends).
- Updates the local `SubscriptionRecord` to reflect `cancel_at_period_end: true`.

### Reactivate — `POST /subscriptions/{user_id}/reactivate`

- Reverses a scheduled cancellation by setting `cancel_at_period_end: false` on Stripe.
- Updates the local `SubscriptionRecord` accordingly.

### Change Tier — `POST /subscriptions/{user_id}/change-tier`

1. Modifies the Stripe Subscription by replacing the current subscription item's price with the new tier's Stripe Price ID.
2. Updates the local `SubscriptionRecord.tier`.
3. Makes an HTTP call to the Permissions Service to sync the user's permissions to the new tier immediately.

## Webhook Processing

The `/webhooks/stripe` endpoint receives raw request bodies and verifies the **Stripe-Signature** header using `STRIPE_WEBHOOK_SECRET` before processing any event. No authentication header is required (the signature serves as authentication).

Seven event types are handled:

### `payment_intent.succeeded`

Updates the matching `Order` (found by `stripe_payment_intent_id`) status to `"confirmed"`.

### `payment_intent.payment_failed`

Updates the matching `Order` status to `"failed"`.

### `customer.subscription.created`

1. Upserts a `SubscriptionRecord` with the subscription's status, period dates, and tier.
2. Syncs the user's permissions by calling the Permissions Service (`POST /subscriptions/{user_id}`).
3. Creates a new `Order` of type `"subscription"` to record the subscription purchase.

### `customer.subscription.updated`

1. Updates the local `SubscriptionRecord` with the latest status, period dates, and `cancel_at_period_end`.
2. If the tier has changed, updates the stored tier and syncs the new tier to the Permissions Service.

### `customer.subscription.deleted`

1. Sets the local `SubscriptionRecord` status to `"expired"`.
2. Syncs the user to the free tier by calling the Permissions Service.

### `invoice.payment_succeeded`

Logged only. No state changes.

### `invoice.payment_failed`

Sets the corresponding `SubscriptionRecord` status to `"past_due"`.

### Tier Extraction from Webhook Payloads

When processing subscription events, the tier is determined by checking, in order:

1. `subscription.metadata.tier`
2. `subscription.items.data[0].price.metadata.tier`
3. Default: `"premium"`

## Integration Points

### Permissions Service

The Commerce Service calls the Permissions Service to keep subscription state synchronized. This is done via HTTP `POST` to:

```
{PERMISSIONS_SERVICE_URL}/subscriptions/{user_id}
```

This call is made on:
- `customer.subscription.created` webhook (activate new tier)
- `customer.subscription.updated` webhook (when tier changes)
- `customer.subscription.deleted` webhook (revert to free tier)
- Subscription tier change via `POST /subscriptions/{user_id}/change-tier`

### API Gateway

All Commerce Service routes are proxied through the API Gateway under the prefix `/api/commerce/*`. The Gateway handles routing and forwards authentication credentials.

### Stripe

Stripe calls `/webhooks/stripe` directly (not through the Gateway) to deliver webhook events. Signature verification ensures event authenticity.

## Dev Tools (Debug Mode Only)

Dev routes are registered only when the `DEBUG` environment variable is `true`. They are intended strictly for local development and testing.

### `POST /dev/simulate-webhook`

Accepts a webhook event payload and calls `handle_event()` directly, bypassing Stripe signature verification. Useful for testing webhook handling without a real Stripe event.

### `GET /dev/webhook-events`

Returns a list of all webhook event types that the service handles. Useful for confirming which events are supported without reading source code.

## MongoDB Indexes

### `commerce_profiles`

| Type | Fields |
|------|--------|
| Unique | `user_id` |
| Sparse Unique | `stripe_customer_id` |

The `stripe_customer_id` index is sparse because newly created profiles may not yet have a Stripe customer assigned.

### `orders`

| Type | Fields |
|------|--------|
| Unique | `order_id` |
| Index | `user_id` |
| Index (descending) | `created_at` |
| Index | `status` |

### `subscription_records`

| Type | Fields |
|------|--------|
| Unique | `user_id` |
| Unique | `stripe_subscription_id` |
| Index | `status` |
